Why can't I just fetch() in the browser?

1. The Browser's Security Guard (CORS)

Browsers enforce a strict rule called Same-Origin Policy. If your app is on localhost:3000 and tries to talk to okta.com, the browser sees a stranger and blocks the call to prevent data theft.

üö´ Blocked: Cross-Origin Request Blocked: The Same Origin Policy disallows reading the remote resource.

2. The "Glass House" Problem

Anything running in the browser (Client Components) is visible to the user. If you put your API_TOKEN in a standard React component, any user can right-click "View Source" and steal your Super Admin credentials. Never do this.

3. The Solution: Server Actions

We use Next.js Server Actions to act as a secure proxy.

  • Browser talks to Server: Your frontend asks your own Next.js server to "Create User". This is allowed (Same Origin).
  • Server talks to Okta: Your Next.js server (running in a secure backend) uses the hidden API Token to call Okta. Servers don't have CORS restrictions.

Deep Dive: What is CORS?

The Analogy: The ID Badge

Imagine your browser is a strict security guard at an office building (Your App). You try to send a messenger to a completely different building across the street (Okta).

The security guard stops the messenger at the door because their ID badge says "Localhost", but the other building only accepts badges from "Trusted Partners".

The Technical Reality

CORS (Cross-Origin Resource Sharing) is a browser security feature. It exists to prevent a malicious website (e.g., evil.com) from making background API calls to your bank (e.g., bank.com) using your logged-in session.

Why it affects this Lab:
Okta's Management API is secure by default. It doesn't know about your local development environment (localhost:3000), so if the browser tries to call it directly, Okta blocks it.

The Fix: We move the request to the Next.js Server (Server Actions). Servers do not have CORS restrictions, so they can pass the request safely.

The Tech Stack (The Tools)

Next.js (The Framework)

What: A production framework built on top of React.
Why: It handles the "boring stuff" like routing (URLs) and backend API proxies automatically. It lets us build a full-stack app in one folder without setting up a separate backend server.

TypeScript (The Safety Net)

What: JavaScript with strict rules.
Why: It prevents bugs before they happen. If you try to access user.firsName (typo), TypeScript yells at you in the editor so you don't break the app during a demo.

Tailwind CSS (The Stylist)

What: A utility-first styling library.
Why: Instead of writing a separate CSS file to center a box, you just write class="flex justify-center". It speeds up UI building by 10x.

NPM (The App Store)

What: Node Package Manager.
Why: It allows us to download pre-written code (libraries) like the Okta SDK or icon sets so we don't have to reinvent the wheel.

Project Map (Where things live)

  • app/ The heart of the app. This folder is your router.
  • page.tsx The Home Page view (localhost:3000/). If you create app/settings/page.tsx, that becomes /settings.
  • layout.tsx The Wrapper. Code here (Navbar, Footer) appears on every page automatically.
  • globals.css Global styles. This is where Tailwind is imported and where you set root font sizes.
  • app/actions/ Server Actions. Functions that run on the server to securely call Okta APIs. This is where we put the Okta API calls so the browser never sees the API Token.
  • app/components/ UI Components. Reusable Lego bricks like <Button /> or <Card /> that we import into pages.
  • app/types/ Interfaces. Definitions of what data looks like (e.g., what an OktaConfig object contains).

Coding Concepts for SEs

React Components

Think of these as Custom HTML Tags. Instead of writing 20 lines of HTML for a user card, you write <UserCard name="Alice" />. It makes code readable and reusable.

Props (Properties)

The Arguments passed to a component. In <UserCard name="Alice" />, "name" is a prop. It allows the same component to display different data.

State (useState)

The component's Short-term Memory. If you type in a text box or toggle a switch, that data is stored in "State". When State changes, the screen updates automatically.
